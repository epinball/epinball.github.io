GamePlan MPU project - Part 1

February 7, 2022

GamePlan manufactured cocktail and upright pinball machines from 1978-1985. The cocktail machines were first, and used the “MPU-1” board. Roger Sharpe was contracted to design their first full/upgright pinball machine SharpShooter which was a huge success for the company. All upright pinball machines used the “MPU-2”.

Regardless of the version of the MPU board, they are very similar. The MPU-1 just had less memory than the MPU-2. Both are Z80 based systems, utilizing now obsolete 4 bit 6551 RAM, ROM space, and an 8255 for the PIA. Other than the obsolete components, the boards suffered from the on-board nicad batteries leaking and causing damage. Most of the time the damage caused could never be recovered to make a reliable board.

Echo Lake had a replacement MPU board available, but it has been awhile since they offered a new run, and I thought this would be a good time to finally give a replacement MPU design a shot.

Goals

  • Use easy to find components as much as possible
    • Use two 74LS377, which is 8bit vs two 74LS379 since they are easier to find
    • Use two 74LS138 vs one 74154
  • Use a Power On Reset w/Watchdog to replace the reset circuitry
  • Use NVRAM to eliminate any need of a battery
  • Have all game roms selectable by a dip switch bank
  • Provide indicators for the different voltage rails
  • Keep the same footprint as the existing MPU-2 board
  • Use through hole to make it easier for soldering
  • Use discrete components vs SIP packages, as I am not too much of a fan of SIP packages (and not as common)

Design

Z80 Microprocessor Circuitry

Of course the MPU-2 schematic is the first place to start. Check out the GP MPU-2 schematic The first page is the Z80 MPU circuitry itself, while the second is the peripheral I/O circuitry. The key part to note is the title block, where the addresses are laid out:

ROM (A/B/C) are addressed 0000-17FF RAM (6810 and the 6551s) are addressed at 8C00-8D7F

Ram/Rom circuitry

Thomas Scherrer’s site z80 info has a very simple Z80 test system found here

Using his design keeps things simple. We will just use a 6264 vs the 55257, as an 8Kx8 RAM is plenty for GamePlan. This will also allow us to use any of the NVRAM products out there in the 6264 configuration.

For the ROM, the largest “Game footprint” is Cyclopes with 3 2732s. That is 12kx8 of Rom space required. A 27128 is 16kx8, which will handle the size of Cyclopes. Utilizing a 4 bank dip switch, we can use a 27C020,a 2Mx8 chip, and have the top 4 Address lines (A14-A17) for selecting what game we are running. This provides space for 16 games.

Watchdog / Reset

For the Watchdog / reset, the Z80 and the CTC are active-low reset, whereas the 8255 is active-high. The MIC1232 is a great fit for this application since it has both as an output. The output of strobe 10/display clock 4 will be used for the watchdog timer reset, just like the MPU-2 had.

Right / Peripheral Interface

The peripheral interfaces are kept the same as the original MPU-2, with two minor exceptions. The Latches used for the Solenoid and Sound ports were 73LS379, which are 4-bit registers and are difficult to find. I have plenty, but why design with the obsolete part? The 74LS377 is 8-bit and easier to find, so this will be used instead.

On the original board a 74154, which is a 4-16 line decoder, is used for driving the strobes. These are difficult to find as well. You can accomplish the same with 2 74138s multiplexed.

KiCad work

I have used KiCad for years now for other projects and love it. It is easy to use, powerful and cross-platform. I did have a hiccup though while designing, in that I started with KiCad 5 and upgraded to KiCad 6 mid-design. I recommend not doing this to avoid a minor headache.

I knew I was going to use OshPark for my proto board set, which OshPark has instructions for using KiCad and what your Design Rules should be. I recommend defining these as soon as possible.

Using KiCad and starting with the schematics, the diagrams were drawn “Atari Style” in using labels to avoid the clutter of traces everywhere. I also tried to group circuits with similar functionality on each sheet. Power pins for each component were put on one sheet as well.

Once the schematic was drawn up, I went through and assigned through hole footprints to each device. Next was running the “Electrical Rules Checker” and getting all of the warnings and errors addressed.

After the schematic was complete and double checked with both the GamePlan MPU schematic as well as Thomas Scherrer’s Z80 system schematic, the netlist was exported to PCBNew.

The mounting holes were first put in place to get an idea of the minimum board size I could make, to keep it as cheap as possible. I also wanted to avoid having to cut any wire ties or make any cable extensions too though, so I didn’t want to move any headers too far.

Components were then placed, with the ratsnest opinionating where components reside. I wanted to keep components facing the same direction as much as possible too, to avoid an IC being installed in the wrong direction.

I had to create a new part for the Display connectors, which are saved in the library with the project.

I defined the common trace sizes to be 0.012in for data and low power use, 0.06in for main power, and 0.05in for med power use.

Keeping VIA count to a minimum, the routing was done manually, assisted with PCBNew’s routing tool to ensure Design Rules are followed.

Once the board was laid out and traces have been routed, the DRC (Design Rule Check) tool was used to identify all items that needed to be fixed. Traces were moved, Components locations were moved etc.

Finally the text for the ROM selection was added.

Combined ROM

To make the combined ROM, all game roms were downloaded and put into separate folders. The roms were combined to make one binary. Each binary was then padded out to a 16k size (like a 27128). Finally each ROM was appended to each other to make up the combined ROM.

I did this with Linux, which Mac would be similar:

  1. Create a 16k file with 0xFF using the tr command. (tr uses octcal numbers, which is why 377):

    dd if=/dev/zero ibs=1k count=16 | tr "\000" "\377" >blank16k.bin

  2. Also lets create a 2mb file with 0xFF as well, doing the same steps above, but 2mb:

    dd if=/dev/zero ibs=1k count=256 | tr "\000" "\377" >blank2mb.bin

  3. For each of the game roms, we will copy the blank16k.bin file as the position holder, and then copy the game rom itself into the position. This way the end result will be the game rom and however many 0xff’s it needs to fulfill the space:

    cp blank16k.bin 1-output.bin
dd if=sharpshooter.bin of=1-output.bin conv=notrunc

cp blank16k.bin 2-output.bin
dd if=globalwarfare.bin of=2-output.bin conv=notrunc

#keep doing this ^ for the remaining roms

  4. Finally, we take all of the padded game roms, combine them, and then copy into our 2mb file from step 2.

  5. Final rom image is stored on github

Bloopers

  • If you notice, that there are a few spots that were skipped in the ROM selection. This was not done on purpose, but since I had it printed on the final board, I might as well stick with it :)

  • I wanted to print out the board so that I can make sure the alignment of the holes and the connectors worked. I needed Legal paper, which I didn’t have any. I ended up finding a copy of my home mortgage and printed on the back of it.

  • I added the LEDs for monitoring the different power rail voltages at the very end and after I already sent the board to OshPark. I saw I was able to cancel without penalty, so I did so and put the LEDs onto the board. When I ran the DRC check, I saw that I actually missed 3 traces! That would have been a big miss. Lesson learned; make sure to check the “unconnected items” tab in the DRC tool!

Conclusion

The Board has been sent out to OshPark, and the parts that I didn’t have in stock have been ordered from Mouser. Next up is building and testing!
